# 分布式存储架构说明

## 一、架构概述

本系统采用**真正分布式存储架构**，支持多节点部署、一致性哈希负载均衡、副本冗余存储、节点间HTTP通信。

**核心特性**：
- ✅ 真正分布式存储（文件存储在选中节点，不是本地）
- ✅ 节点间HTTP通信（内部存储接口）
- ✅ Nginx负载均衡（请求分发）
- ✅ 统一分片逻辑（所有文件都走分片流程）
- ✅ 分布式读取（从目标节点读取文件）

## 二、核心概念

| 概念 | 说明 |
|------|------|
| **StorageNode（存储节点）** | 代表一个独立的存储服务实例，包含主机地址、端口、状态等信息 |
| **base-path（本地存储路径）** | 当前实例的物理存储目录，如 `./storage` |
| **ConsistentHash（一致性哈希）** | 根据文件key选择存储节点的算法，支持虚拟节点均衡分布 |
| **Zookeeper** | 分布式协调服务，用于节点注册、心跳检测、故障发现 |
| **Redis** | 分布式缓存，用于上传进度跟踪、元数据缓存、分布式锁 |

## 四、文件上传流程（统一分片逻辑）

### 4.1 设计原则

**所有文件上传统一走分片逻辑**：
- 小文件：`chunkCount=1`，单分片
- 大文件：`chunkCount>1`，多分片

**前置条件**：必须先注册存储节点，否则上传服务不可用。

### 4.2 上传流程

```
用户上传文件
    ↓
检查节点可用性（无节点则返回 503）
    ↓
计算分片数量：chunkCount = ceil(fileSize / chunkSize)
    ↓
初始化上传：
    ├─ 生成 fileKey
    ├─ 生成 AES 密钥
    └─ 缓存上传信息到 Redis
    ↓
上传所有分片（循环）：
    ├─ 加密分片数据
    ├─ ConsistentHash.selectNodes(fileKey + "-chunk-" + index)
    ├─ 选择存储节点
    ├─ LocalStorageService.store() → 写入存储目录
    ├─ 记录分片信息到 file_chunk 表（含 storageNodeId）
    └─ 写入副本（可选）
    ↓
完成上传：
    ├─ 创建 file_metadata 记录
    ├─ 更新分片的 fileId
    └─ 缓存元数据到 Redis
```

### 4.3 前端交互

| 文件类型 | 前端行为 | 后端处理 |
|---------|---------|---------|
| **小文件**（< chunkSize） | 调用 `/file/upload` | 后端内部统一分片（chunkCount=1） |
| **大文件**（> chunkSize） | 前端分片上传，显示进度 | 每个分片独立处理 |

**用户体验优化**：
- 小文件直接上传，无进度对话框
- 大文件显示分片进度，用户可感知上传状态

### 4.4 数据存储结构

| 表 | 字段 | 说明 |
|---|---|---|
| **file_metadata** | chunkCount | 分片总数（小文件=1） |
| **file_chunk** | storageNodeId | 分片存储的主节点ID |
| **file_chunk** | replicaNodeIds | 副本节点ID列表 |
| **file_chunk** | storagePath | 分片存储路径 |

所有分片信息统一存储在 `file_chunk` 表，便于管理和迁移。

## 五、一致性哈希算法

### 5.1 原理

一致性哈希将节点映射到一个哈希环上，根据数据的哈希值顺时针查找最近的节点。

```
哈希环示意图：

        0
        │
   Node1├───────Node2
        │       │
        │       │
   Node3├───────┘
        │
   2^32-1

虚拟节点：每个物理节点对应150个虚拟节点，均匀分布在环上
```

### 5.2 节点选择逻辑

```java
// 根据 fileKey + chunkIndex 计算哈希
String hashKey = fileKey + "-chunk-" + chunkIndex;
long hash = FNV1_32_HASH(hashKey);

// 顺时针查找最近的节点
Map.Entry<Long, Long> entry = hashRing.ceilingEntry(hash);
if (entry == null) {
    entry = hashRing.firstEntry(); // 环形，回到起点
}
StorageNode node = nodeMap.get(entry.getValue());
```

### 5.3 副本存储

系统支持多副本存储（配置 `replica-count: 3`），同一文件的多个副本分布在不同节点：

```java
// 选择3个不同节点存储副本
List<StorageNode> nodes = consistentHash.selectNodes(hashKey, 3);
```

## 六、节点注册与管理

### 6.1 节点注册

```bash
# 注册节点
POST /api/node/register?nodeName=node1&host=localhost&port=8080
```

注册流程：
1. 在 Zookeeper 创建临时节点 `/nodes/node1`
2. 数据库 `sys_storage_node` 表插入记录
3. ConsistentHash 哈希环添加节点（含150个虚拟节点）

### 6.2 心跳检测

系统每10秒检测节点状态：
- Zookeeper 临时节点存在 → 节点在线
- Zookeeper 临时节点消失 → 节点离线，更新数据库状态

### 6.3 故障转移

节点离线时：
1. ConsistentHash 自动移除节点
2. 新上传的文件自动路由到其他在线节点
3. 已存储的文件副本在其他节点仍可访问

## 十、常见问题

### Q1：为什么必须先注册节点才能上传？

系统采用统一分片逻辑，所有文件（包括小文件）都需要通过一致性哈希选择存储节点。无节点时上传服务不可用，返回 503 错误。

### Q2：只有一个节点时，分片都在这个节点上吗？

是的。一致性哈希根据 `fileKey + "-chunk-" + index` 计算哈希值，只有一个节点时，所有分片都会落在该节点。

### Q3：如何实现真正的分布式存储？

1. 部署多个服务实例在不同机器
2. 每个实例启动时自动注册到 Zookeeper
3. 配置副本数 > 1，分片自动分布到不同节点
4. 节点故障时，副本在其他节点仍可访问

### Q4：节点离线后文件还能访问吗？

- 如果文件有副本在其他在线节点 → 可以访问
- 如果文件只存储在离线节点 → 暂时无法访问，需等待节点恢复

### Q5：小文件为什么也走分片逻辑？

统一分片逻辑的好处：
1. **数据结构一致**：所有文件分片信息都在 `file_chunk` 表
2. **节点追踪**：每个分片都有 `storageNodeId`，便于迁移和故障恢复
3. **代码简洁**：无需区分大小文件的处理逻辑
4. **分布式改造**：只需改造存储服务层，无需重构上传逻辑
